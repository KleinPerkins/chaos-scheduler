## Learned User Preferences

- chaos-scheduler is primarily a personal app for the user and a few friends; skip enterprise-grade hardening/steps that add no functional value at that scale unless there is a specific loss of functionality (e.g. accept documented low-impact release-pipeline residuals rather than over-engineering fixes for them).
- When the local working tree at ~/dev/personal/chaos-scheduler must stay pristine, land repo changes via the GitHub git-data API (create blobs → tree → commit → ref) as an atomic single-commit Conventional-Commits PR, rather than local git.
- Solve "owner can't self-approve their own PR" by using a dedicated GitHub App auto-merge bot to satisfy the branch-protection review gate, rather than lowering the required-review gate.
- Chose a passwordless Tauri minisign signing key; do releases/updates unattended without a key password prompt.
- Don't declare a session/build "done" while the working tree is dirty or spawned worktrees/branches linger — as the build agent, reconcile every worktree you spawned, land all work (commit/push/merge), sync local+remote, and prune merged remote branches; reconcile or hand off any other-repo work you touched via a copy/paste prompt to a fresh agent. Don't over-claim "clean".
- Prefer one-concern-per-PR (independent review/revert/bisect + a per-concern failing-first test) over bundling multiple concerns into one diff; bundling is only an expediency, never the quality bar.
- Prevent SDK/MCP/integration testing from polluting the production scheduler DB; keep Playwright e2e smoke tests and UAT as required test surfaces.
- Prefer a Figma design/prototyping phase before implementing UX changes — UX shouldn't be a black box only visible after the build ships; iterate on Figma mocks (point-at-a-layer + written instructions via the local figma-desktop MCP) before coding.

## Learned Workspace Facts

- chaos-scheduler (KleinPerkins/chaos-scheduler, public) is a Tauri 2 desktop/tray app — Rust backend (src-tauri/) + React/TypeScript frontend (src/) with SQLite persistence — being decoupled and rebranded off of chaos-labs.
- Core model (shipped 0.6.0): `environment` + `managed_externally` replaced overloaded `corpus`; `SchedulerService` is the business-logic boundary with Tauri IPC, REST `/api/v1`, SDK, and MCP as thin adapters; bidirectional Cursor integration via MCP server + `cursor_agent` Cloud Agents operator.
- Release/update pipeline: `release.yml` is a REUSABLE workflow invoked by `release-please.yml` via `workflow_call`, running in a required-reviewer `release` GitHub Environment that holds the Tauri minisign signing key (public key in src-tauri/tauri.conf.json), npm, and Apple Developer-ID secrets; it builds/signs+notarizes the DMG, uploads minisign-signed updater artifacts, and serves `latest.json` via GitHub Releases (a `workflow_dispatch` path exists to re-attach assets to an existing tag). Build/publish jobs are gated on release-please per-component `*--release_created` outputs PLUS `if:`/`workflow_call` inputs (build_desktop/publish_sdk/publish_mcp) — a mis-evaluated upstream gate can create tags/releases yet skip all build/publish (leaving `latest.json` 404, npm unpublished, and `pending_deployments` empty because nothing ever reached the environment gate). release-please assigns GitHub's "Latest" flag non-deterministically, so an unconditional `gh release edit "$desktop_tag" --latest` step keeps the `releases/latest/download/latest.json` updater endpoint pointed at the desktop tag. The minisign updater signing is separate from and in addition to Apple signing (neither is skipped for shipped releases); losing the minisign private key permanently breaks auto-update. Client side, the app does consent-gated background/on-launch update checks surfaced via an in-app affordance (banner + tray popup row + skip-version), with Rust as the source of truth and best-effort IPC events hydrating both windows.
- `main` is protected (ruleset 18513148: PR required, strict `ci-required`, linear history, 1 approving review); auto-merge via GitHub App bot `chaos-scheduler-automerge` (AUTOMERGE_APP_ID + AUTOMERGE_APP_PRIVATE_KEY; App approval satisfies the review gate).
- Git hooks are active via lefthook: commit-time prettier + commitlint (Conventional Commits required; never use --no-verify); pre-push runs tsc + cargo test, filtered by changed-file globs.
- `gh` is authenticated for KleinPerkins/chaos-scheduler.
- Desktop ships as a universal DMG (Apple Developer-ID signed + notarized; auto-update via `releases/latest/download/latest.json`) plus separately-versioned npm `@chaos-scheduler/sdk` and `@chaos-scheduler/mcp-server`. MCP/SDK distribution is APP-OWNED provisioning — NOT DMG-bundled and NOT floating `npx`: the desktop app is the lifecycle owner and provisions the existing npm packages into an app-managed dir at a version pinned to the build (first-run/opt-in via the Integrations screen, since macOS DMGs are drag-install with no installer script), with Node the only external runtime dep and graceful degradation if missing (mcp.rs is the ~2200-line managed provisioner: install/repair/uninstall, mutex recovery, orphaned-dir cleanup, mcp.json merge/backup). mcp-server is tsup-bundled for a zero-npm-footprint (transitive deps like @modelcontextprotocol/sdk + zod inlined) but `@chaos-scheduler/sdk` MUST stay `external` (never in tsup `noExternal`) or independent SDK releases break. Release ordering gate: sdk-ts → mcp-server → clean-install/CLI smoke → desktop build pins MCP version → upload updater assets → re-pin GitHub Latest → smoke `latest.json`.
- Tauri crate/JS parity: the `tauri` Rust crate and `@tauri-apps/api` must stay major.minor-aligned or the release `tauri build` fails (root cause of the 0.3.0 desktop failure — CI ran `cargo build` + the JS build but never `tauri build`; fixed by pinning the crate to 2.11.5). A compile-free lockfile parity gate `scripts/check-tauri-versions.mjs` + a `tauri-versions` job wired into the `ci-required` fan-in now blocks crate/api major.minor drift on every PR.
- Cursor-managed worktrees created via the Agents Window Worktree dropdown may NOT register in the primary checkout's `git worktree list` (nor show under `~/.cursor/worktrees` / AWA disk scans); they surface only in the Agents Window Worktree dropdown, so verify there (not just git) when reconciling leftover chaos-scheduler worktrees. By contrast, worktrees created via the `dev-worktree.sh ensure` helper live under `~/.local/share/awa/worktrees/chaos-scheduler/…` and DO appear in the primary checkout's `git worktree list`; a linked worktree cannot fast-forward the primary checkout's `main` (git refuses to move a branch checked out elsewhere), so sync the primary in its own tree (`git -C ~/dev/personal/chaos-scheduler pull --ff-only`). Parallel build tracks must keep disjoint file sets (Rust / sdk-ts+INTEGRATION+cursor-rule / mcp-server) and Rust work stays a single sequential writer (one-writer-per-worktree).
- `.cursor/hooks/state/continual-learning-index.json` is gitignored (via `~/.gitignore_global`) while `AGENTS.md` is tracked; the continual-learning AGENTS.md delta is default-deny and gets published to `main` via the git-data API (blob→tree→commit→ref) single-commit PR pattern, keeping the working tree undisturbed.
- Design iteration happens in Figma against the Affirm "Project Genome" design system (dark mode) — the app's "Home / Mission Control" screen is mocked in Figma (fileKey twQmWC8dWT4tqeqIigNsRy). Two Figma MCP servers exist: the remote "figma" (CANNOT read live selection) and the local "figma-desktop" (`http://127.0.0.1:3845/mcp`, requires the Figma desktop app open + port 3845 listening) which CAN read `figma.currentPage.selection` — use figma-desktop for point-at-a-layer-and-instruct iteration, not the remote server.
- Live REST API/SDK/MCP integration tests hit `http://127.0.0.1:9618` and write the production SQLite DB (`~/Library/Application Support/com.chaosscheduler.app/scheduler.db`); Playwright e2e uses `mockIPC` and does not. Default environments: **`production`** (UI + API) and **`sandbox`** (MCP writes / UAT). MCP protects `production` by default (`CHAOS_SCHEDULER_MCP_PROTECTED_ENVIRONMENTS`); backend protection defaults empty so the UI/API can manage production workflows. Schema v11 migrates legacy `source`/`instance` names away.
