## Learned User Preferences

- chaos-scheduler is primarily a personal app for the user and a few friends; skip enterprise-grade hardening/steps that add no functional value at that scale unless there is a specific loss of functionality (e.g. accept documented low-impact release-pipeline residuals rather than over-engineering fixes for them).
- When the local working tree at ~/dev/personal/chaos-scheduler must stay pristine, land repo changes via the GitHub git-data API (create blobs → tree → commit → ref) as an atomic single-commit Conventional-Commits PR, rather than local git.
- Solve "owner can't self-approve their own PR" by using a dedicated GitHub App auto-merge bot to satisfy the branch-protection review gate, rather than lowering the required-review gate.
- Chose a passwordless Tauri minisign signing key; do releases/updates unattended without a key password prompt.

## Learned Workspace Facts

- chaos-scheduler (KleinPerkins/chaos-scheduler, public) is a Tauri 2 desktop/tray app — Rust backend (src-tauri/) + React/TypeScript frontend (src/) with SQLite persistence — being decoupled and rebranded off of chaos-labs.
- Core rearchitecture: replace the overloaded `corpus` ("source"|"instance") with `environment` plus a separate `managed_externally` flag; extract a single `SchedulerService` business-logic boundary with Tauri IPC, REST `/api/v1`, SDK, and MCP as thin adapters; provide bidirectional Cursor integration (MCP server + Cursor Cloud Agents).
- Release/update pipeline: Tauri v2 auto-updater signed with a minisign key at ~/.tauri/chaos-scheduler.key (public key lives in src-tauri/tauri.conf.json); a `release` GitHub Environment holds TAURI_SIGNING_PRIVATE_KEY (+ _PASSWORD); `latest.json` served via GitHub Releases. Losing the private key permanently breaks auto-update for installed users. This minisign key signs the updater artifacts only — separate from and in addition to the Apple Developer-ID code-signing + notarization applied to the shipped desktop DMG (Apple signing is NOT skipped for shipped releases).
- `main` is protected by ruleset 18513148: PR required, strict `ci-required` status check, linear history, non-fast-forward, `required_approving_review_count: 1`, `require_code_owner_review: false`, empty bypass_actors.
- Auto-merge runs via a GitHub App bot (`chaos-scheduler-automerge`) using repo secrets AUTOMERGE_APP_ID + AUTOMERGE_APP_PRIVATE_KEY (.github/workflows/app-auto-merge.yml); the App's approval satisfies the review gate. The interim GITHUB_TOKEN-based auto-merge.yml was retired.
- Git hooks are active via lefthook: commit-time prettier + commitlint (Conventional Commits required; never use --no-verify); pre-push runs tsc + cargo test, filtered by changed-file globs.
- `gh` is authenticated for KleinPerkins/chaos-scheduler.
- Shipped as of Jul 2026: npm `@chaos-scheduler/{sdk,mcp-server}@0.3.0` (versioned separately from desktop) and desktop `0.3.1` — a universal DMG (`Chaos.Scheduler_0.3.1_universal.dmg`) that is Apple Developer-ID signed AND notarized (`spctl` = accepted / Notarized Developer ID, no Gatekeeper bypass on install); the updater endpoint `https://github.com/KleinPerkins/chaos-scheduler/releases/latest/download/latest.json` returns 200 → v0.3.1 (auto-update live). MCP stdio server launches via `npx -y @chaos-scheduler/mcp-server`.
- Tauri crate/JS parity: the `tauri` Rust crate and `@tauri-apps/api` must stay major.minor-aligned or the release `tauri build` fails (root cause of the 0.3.0 desktop failure — CI ran `cargo build` + the JS build but never `tauri build`; fixed by pinning the crate to 2.11.5). A compile-free lockfile parity gate `scripts/check-tauri-versions.mjs` + a `tauri-versions` job wired into the `ci-required` fan-in now blocks crate/api major.minor drift on every PR.
- release-please's multi-component setup assigns GitHub's "Latest" flag non-deterministically — the assetless `chaos-scheduler-tauri-v*` component tag can grab it and break the `releases/latest/download/latest.json` updater endpoint; durable fix is an unconditional `gh release edit "$desktop_tag" --latest` step in the `build-macos` job of `release.yml` (residual per docs/RELEASING.md: an npm-only sdk/mcp release with no desktop bump won't run it — low impact).
